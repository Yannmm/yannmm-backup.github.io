发起 Pull Request

Posted on 2019-08-11 | In 代码

工作原理 / 实例 / 下一步

Pull Request 极大的方便了我们在 Bitbucket 上进行开发协作。通过网页展示待合并修改及相关讨论，大大提升了用户体验。

简单来说，Pull Request 是一种通知机制，提醒相关人员功能已经开发完毕。分支上的工作完成后，通过 Bitbucket 账户发起 Pull Request，请求代码审查，进而合并功能。

但是 Pull Request 绝不仅限于通知 —— 它更是一种组织讨论的形式。团队成员可以提问，给出建议和反馈，甚至直接作出修改，所有活动都记录在案。

与其他合作模式相比，Pull Request 的工作流程更为线性。虽然 SVN 和 Git 都能够借助脚本自动发送通知邮件；但讨论只能通过邮件进行，此时如果继续提交修改，经常会让人感到迷惑。Pull Request 使用网页组织信息和功能，入口就在仓库旁边，大大提升了用户体验。

Pull Request 的组成

Pull Request 意味请求其他开发者（如项目维护者）将我们开发完成的代码（分支）合并到他所管理的仓库。为此，必须提供4条关键信息：原仓库，原分支，目标仓库，目标分支。

Bitbucket通常能够自动填写这些信息。但不同合作流程的信息可能变化。上图表示将新功能合并至官方仓库的 master 分支，但 Pull Request 的用法远不止于此。

工作原理

Pull Request 适用于以下工作流程：Feature Branch Worflow，Gitflow Workflow，Forking workflow。基于前提是分支或仓库不同，所以 Pull Request 并不适用于 Centralized Workflow。不同流程下 Pull Request 的使用方式不一样，但大致可以归纳为：

  1. 开发者在本地分支上实现功能；
  2. 开发者推送本地分支到公共仓库；
  3. 开发者使用 Bitbucket 提交 PR；
  4. 其他人审查代码，提出意见，做出修改；
  5. 项目维护者合并代码至官方仓库，随后关闭 Pull Request。

下面我们逐一说明 Pull Request 在各种流程下的使用。

Feature Branch Workflow 与 Pull Requests

所有开发者利用同一个共享仓库进行协作，基于功能创建分支。开发完成后提交 Pull Request，围绕功能展开讨论，最后将代码合并至 master 分支。

由于只有一个公共仓库，所以 Pull Request 的两个仓库相同。一般来说，功能分支为原分支，master 为目标分支。

项目维护者必须在收到 Pull Request 后采取行动：功能符合要求，合并然后关闭 Pull Request；如果遇到问题，则给出反馈，后续修改将随后显示。

还可以在功能完成前开启 Pull Request。举例来说，假设我们在开发过程中遇到困难，就可以提前开启包含当前工作进度的 Pull Request，邀请其他开发者提供建议，甚至直接修改代码。

Gitflow Workflow 与 Pull Requests

流程与 Feature Branch Workflow 类似，但针对项目发布制定了严格的分支策略，借助 Pull Request 讨论项目的发布与维护。


Pull Request 的作用与之前一样，通知团队成员分支代码需要审核。

功能代码一般合并至开发分支；发布和修复代码可以合并至开发 或 master 分支。Pull Request 的作用就在于正式管理这些合并。

Forking Workflow 与 Pull Requests

此流程要求开发者将已完成的功能推送至自己的公共仓库，而非共享仓库。然后提交 Pull Request ，通知项目维护者审核。

Pull Request 在此流程中扮演非常重要的角色，这是项目维护者了解他人开发工作的唯一途径。

由于每位开发者都有自己的公共仓库， 此时显然 Pull Request 的两个仓库不同。原仓库对应开发者的公共仓库，原分支对应功能分支。假设我们想让自己的修改成为基础代码的一部分，则目标仓库对应官方仓库，目标分支对应 master 分支。

第三方开发者之间还可以通过 Pull Request 展开协作。假设我们正在与队友合作开发，基于同一个功能分支，针对对方仓库发起 Pull Request （而非主仓库）即可。

两位开发者利用 Pull Request 沟通，共同开发。功能完成后，其中一个人再发起一个 Pull Request，请求合并到官方仓库。由此可见 Pull Request 的强大，是不可或缺的协作工作。

举个栗子 🌰

下面的例子将着重讲解 Pull Request 在 Forking workflow 中的运用，同样适用于小规模团队及对贡献开源项目。

Mari 是开发者，John 是项目维护者。它们都有自己的公共仓库，其中 John 的是官方仓库。

Mary fork 项目

Mary 首先必须 fork john 的仓库才能进行开发。登陆 Bitbucket，找到仓库，点击 fork 按钮即可。

只需填写名字和描述，她就在 Bitbucket 服务器上建立了自己的项目仓库。

注意，git clone 将远程仓库复制到本地，并自动创建名为 origin 的指针，指向对应的（Mary）远程仓库。

Mary 开发功能

Mary 在本地创建功能分支，着手开发。同时这个分支会是将来 Pull Request 中的原分支。

开发过程中 Mary 可以尽情提交 (commit)。如果提交历史看起来很混乱，完全可以使用 interactive rebase 命令移除或合并提交。这对于大规模项目意义重大，清晰的历史记录意味着项目维护者能够轻易读懂开发者的意图。

Mary 推送功能至远程仓库

开发完成后，Mary 使用 git push 命令将本地分支推送到自己的远程仓库（而非官方仓库）。

这使得其他开发人员（如项目维护者）能够访问她的代码。

Mary 发起 Pull Request

推送后，Mary 可以通过 Bitbucket 账户发起 Pull Request：找到自己的远程仓库， 点击右上角的 Pull Request 按钮。当前仓库默认为原仓库，Mary 需要指定原分支，目标仓库和目标分支。

要想将修改成为基础代码的一部分，Mary 需要将原分支指定为自己的功能分支，指定目标仓库是 john 的官方仓库， 目标分支指定为 master 分支。此外还需要填写标题和描述。除了 John 以外，Mary 还可以邀请将除了 John 以外的任何人审阅代码，只需将他们添加为 Reviewer 即可。

Pull Request 一旦创建，通知会以邮件和 Bitbucket 消息的形式发送给 John。

John 审阅 Pull Request

John 通过点击自己 Bitbucket 仓库的页面上的 Pull request 标签，查看所有的 Pull Request。再次点击 Mary 的 Pull Request，具体描述，提交记录及所有改动一览无余。

如果一切顺利，John 只需点击 Merge 按钮，将修改合并至 master 分支即可。

但假设 John 发现了一个小 bug，希望 Mary 在合并之前修复。那么他可以针对整个 Pull Request 或特定提交记录做出评论。

Mary 追加提交

对于任何疑问，Mary 都可以在 Pull Request 中反馈，将其作为讨论组使用。

Mary 在功能分支修改并提交，修正错误，推送到自己的远程仓库，正如她之前做的一样。这次提交被自动追加到之前的 Pull Request 中，显示在先前 John 的评论之后，便于再次审阅。

John 接受 Pull Request

终于，John 接受修改，将代码合并至 master 分支，关闭 Pull Request。至此新功能被正式添加到项目中，其他开发者可以通过 git pull 命令获取更新。

下一步

你已经掌握了使用 Pull Request 所需的一切知识，是时候将它融入自己的工作流程当中了。记住，Pull Request 不是 Git-based collaboration workflows 的代替品。而是在此之上，让团队协作更容易的催化剂。
