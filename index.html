容器视图控制器

Posted on 2019-06-24 | In 开发

缘起

作为一名 iOS 开发者， “UIViewController 到底是干嘛的？” 这个问题一直让我感到困惑。说来惭愧，早已不是新手的我不但没能搞懂这个问题，更经常拿“会用就行，细节不重要”之类的话安慰自己，不求甚解，得过且过。直到最近，又一次因为项目需要，必须自行实现容器控制器。经过一番搜索，查阅之后，终有所感悟，特此记录。

组合

同类对象组合，形成特定结构，是 UIKit 中常见的设计模式。例如，UIView 组成 view hierarchy，UIResponder 组成 responder chain。类似的，视图控制器之间也可以互相组合，有两种方式：

  1. modal presentation - 基于当前视图，展示另一个控制器的视图，阻断下层视图的交互
  2. 容器控制器 - 一个控制器同时持有一个或多个子控制器，将它们的视图据为己用

说到这里，很多人会有一个疑问：为什么（要组合控制器）？有什么好处？如果单纯只是为了借用其他控制器的视图，直接加入当前视图结构不就行了？

然而，除开管理视图，控制器还有许多重要职责。其中最具代表性的，当属转发各种视图事件：

  • viewWillAppear
  • viewDidAppear
  • viewWillDisappear
  • viewDidDisappear

这些方法对于开发的重要性不言而喻。仅将视图加入当前视图结构，是不会触发这些方法的。

今天我们将探讨第二种组合方式：容器控制器。思考 UIKit 是如何实现这类控制器的；自己实现时如何组织思路，要注意哪些细节；最后，回答开篇的问题： “UIViewController 到底是干嘛的？” 。

UINavigationController

UINavigationController 是 UIKit 中最具代表性的容器控制器（当然还有 UITabBarController）。或许我们早已习惯它的存在，本着够用就好的原则，很少考虑其实现原理。换句话说，如果让你自行实现一个导航控制器，要解决哪些问题呢？例如：

  • 如何处理导航栏与子控制器关系的？
  • 如何触发子控制器的视图事件？
  • 两个子控制器之间如何过渡？
  • 如何实现手势返回？

当然，本文的目的不是回答上述问题。而是想强调没有什么黑魔法，只要我们愿意，一样可以实现功能极为复杂的容器控制器。

套路

首先必须明确，容器控制器的目标在于建立两套联系：

  1. 父控制器与子控制器之间的联系
  2. 视图结构

从功能出发，容器控制器必须能够添加、删除子控制器，这里给出如下套路，照搬即可：

添加

  1. 建立父子关系。注意，这个方法首先自动调用子控制器的willMoveToParentViewController:，通知其即将加入父控制器；
  2. 将子控制器视图加入自己的视图结构；
  3. 通知子控制器，它已加入父控制器；

删除

  1. 通知子控制器，其即将被移出父控制器；
  2. 将子控制器视图移出视图结构；
  3. 解除父子关系。注意，这个方法最后调用didMoveToParentViewController，通知子控制器父子关系已经解除。

转发视图事件

UIKit 不推荐直接调用子控制器的视图事件方法，而是使用 beginAppearanceTransition:animated: 和 endAppearanceTransition。我们需要基于自定义容器控制器的显示特点，在合适的地方调用这两个方法。显然，这将大大增加工作量。所幸，UIKit 提供 shouldAutomaticallyForwardAppearanceMethods 自动转发视图事件。 但其工作原理并不透明。手动转发 vs. 自动转发，孰优孰劣暂时无法给出结论， 有机会的我们将在另一片文章中比较。

视图控制器的责任

现在，我想回答最初的问题：”UIViewController 到底是干嘛的？”

搞清这个问题其实很简单：打开苹果文档，找到 UIViewController Reference，查看它的 topics，每个 topic 都是它的一项职责，包括但不限于：

  • segue - 转场
  • view events - 视图事件
  • modal presentation - 阻断式显示
  • safe area - 安全区域
  • rotation - 屏幕旋转
  • 3D touch
  • status bar
  • containment - 容器控制器

不难看出，上述职责或多或少都与显示有关，但又不是纯显示，所以下放给 UIView 是不合适的。能够履行这些职责，并根据要求操作视图的对象，才是合理人选。视图控制器这个名字，完美阐释了它的定位，苹果在设计模式方面的功底可见一斑。

  • View Controller Programming Guide for iOS

URL 概念与比较

Posted on 2019-04-23 | In 代码

URL与URI

说到URL，必须从URI讲起。

出于 hpertext 的需要，人们需要一种准确的方法表示资源。URL和URN分别从各自的维度定义资源：获取的方式 / 名称。二者属于 URI 的不同子概念。

关于URL与URN的区别，简单来说就是：URN可以是一个人的身份证号；而URL是这个人的住址。

个人理解，相较于上述经典定义，目前对URL的使用已经升华为表示一个动作，如获取资源，执行任务（如查询，这是我们每天都会使用的）。

注意

不要纠结于URL与URI到底有什么区别，完全没有必要。根据W3C的最新定义，URL和URI广义上的意义相同，即代表资源的字符串。所以用哪个都行，但在编程语言中，前者使用较多。

下面讨论的各项URI的性质，都适用于URL。

语法

URI的完整语法形式如下：

  • scheme，规定URI剩余部分的语法。常见的scheme有如http / https / ftp / mailto / file / data / irc。注意，虽然大部分scheme都有同名protocol与之对应（这是其设计之初的用途），但例外是存在的，如file。
  • 双反斜线（//）。如果authority不存在，则path不应以包含//。
  • authority，可以由下述部分组成：
    • authentication，包含用户名和密码，用于验明身份；
    • host，可以是域名，ip地址或任意表示作用域的字符串；
    • 端口号，如果host是域名或ip地址时。
  • 路径，类似于文件路径（但不等同），以层级的形式定义，层级之间用/分隔，以/开头。（注意，根据这里的定义，加上之前的//，会变成///。这是正常的，但显示时，还是会变成//）
  • query，表示查询的内容。以?开头，单个查询的形式为key=value，多个查询用&分隔。
  • fragement，表示当前URI所指内容的一部分。以#开头。例如，如果当前URI指向一篇文章，则#Summary指这个文件的Summary部分。

URI中的字符可以是ASCII字符中的英文，数字，-，.，_，~。其他字符必须百分号转义。

注意

~在URI中是不需要转义，这意味着我们可以使用它定义path，代表文件路径。这种用法实际上来自Unix：~代表当前用户的home directory，是一种惯例。使用URL表示文件路径时，要格外注意。

绝对与相对

所谓绝对URI指具有完整形式的URI；相对URI指URI reference。

URI reference的形式包括但不限于：

  • 完整URI：http://example.org/wiki/Main_Page
  • 不包括scheme：example.org/wiki/Main_Page
  • 只有部分path：wiki/Main_Page
  • 空：nothing here -_-

使用时，根据特定算法（RFC 3986），将URI reference与base URI合并，从而得出一个完整URI。很明显，如果URI reference本身就是一个完整URI，则base URI不起作用。
