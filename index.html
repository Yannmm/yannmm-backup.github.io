发起 Pull Request

发表于 2019-08-11 | 分类于 代码

工作原理 / 实例 / 下一步

Pull Request 极大的方便了我们在 Bitbucket 上进行开发协作。通过网页展示待合并修改及相关讨论，大大提升了用户体验。

简单来说，Pull Request 是一种通知机制，提醒相关人员功能已经开发完毕。分支上的工作完成后，通过 Bitbucket 账户发起 Pull Request，请求代码审查，进而合并功能。

但是 Pull Request 绝不仅限于通知 —— 它更是一种组织讨论的形式。团队成员可以提问，给出建议和反馈，甚至直接作出修改，所有活动都记录在案。

与其他合作模式相比，Pull Request 的工作流程更为线性。虽然 SVN 和 Git 都能够借助脚本自动发送通知邮件；但讨论只能通过邮件进行，此时如果继续提交修改，经常会让人感到迷惑。Pull Request 使用网页组织信息和功能，入口就在仓库旁边，大大提升了用户体验。

Pull Request 的组成

Pull Request 意味请求其他开发者（如项目维护者）将我们开发完成的代码（分支）合并到他所管理的仓库。为此，必须提供4条关键信息：原仓库，原分支，目标仓库，目标分支。

Bitbucket通常能够自动填写这些信息。但不同合作流程的信息可能变化。上图表示将新功能合并至官方仓库的 master 分支，但 Pull Request 的用法远不止于此。

工作原理

Pull Request 适用于以下工作流程：Feature Branch Worflow，Gitflow Workflow，Forking workflow。基于前提是分支或仓库不同，所以 Pull Request 并不适用于 Centralized Workflow。不同流程下 Pull Request 的使用方式不一样，但大致可以归纳为：

  1. 开发者在本地分支上实现功能；
  2. 开发者推送本地分支到公共仓库；
  3. 开发者使用 Bitbucket 提交 PR；
  4. 其他人审查代码，提出意见，做出修改；
  5. 项目维护者合并代码至官方仓库，随后关闭 Pull Request。

下面我们逐一说明 Pull Request 在各种流程下的使用。

Feature Branch Workflow 与 Pull Requests

所有开发者利用同一个共享仓库进行协作，基于功能创建分支。开发完成后提交 Pull Request，围绕功能展开讨论，最后将代码合并至 master 分支。

由于只有一个公共仓库，所以 Pull Request 的两个仓库相同。一般来说，功能分支为原分支，master 为目标分支。

项目维护者必须在收到 Pull Request 后采取行动：功能符合要求，合并然后关闭 Pull Request；如果遇到问题，则给出反馈，后续修改将随后显示。

还可以在功能完成前开启 Pull Request。举例来说，假设我们在开发过程中遇到困难，就可以提前开启包含当前工作进度的 Pull Request，邀请其他开发者提供建议，甚至直接修改代码。

Gitflow Workflow 与 Pull Requests

流程与 Feature Branch Workflow 类似，但针对项目发布制定了严格的分支策略，借助 Pull Request 讨论项目的发布与维护。


Pull Request 的作用与之前一样，通知团队成员分支代码需要审核。

功能代码一般合并至开发分支；发布和修复代码可以合并至开发 或 master 分支。Pull Request 的作用就在于正式管理这些合并。

Forking Workflow 与 Pull Requests

此流程要求开发者将已完成的功能推送至自己的公共仓库，而非共享仓库。然后提交 Pull Request ，通知项目维护者审核。

Pull Request 在此流程中扮演非常重要的角色，这是项目维护者了解他人开发工作的唯一途径。

由于每位开发者都有自己的公共仓库， 此时显然 Pull Request 的两个仓库不同。原仓库对应开发者的公共仓库，原分支对应功能分支。假设我们想让自己的修改成为基础代码的一部分，则目标仓库对应官方仓库，目标分支对应 master 分支。

第三方开发者之间还可以通过 Pull Request 展开协作。假设我们正在与队友合作开发，基于同一个功能分支，针对对方仓库发起 Pull Request （而非主仓库）即可。

两位开发者利用 Pull Request 沟通，共同开发。功能完成后，其中一个人再发起一个 Pull Request，请求合并到官方仓库。由此可见 Pull Request 的强大，是不可或缺的协作工作。

举个栗子 🌰

下面的例子将着重讲解 Pull Request 在 Forking workflow 中的运用，同样适用于小规模团队及对贡献开源项目。

Mari 是开发者，John 是项目维护者。它们都有自己的公共仓库，其中 John 的是官方仓库。

Mary fork 项目

Mary 首先必须 fork john 的仓库才能进行开发。登陆 Bitbucket，找到仓库，点击 fork 按钮即可。

只需填写名字和描述，她就在 Bitbucket 服务器上建立了自己的项目仓库。

Mary clone 项目到本地

注意，git clone 将远程仓库复制到本地，并自动创建名为 origin 的指针，指向对应的（Mary）远程仓库。

Mary 开发功能

Mary 在本地创建功能分支，着手开发。同时这个分支会是将来 Pull Request 中的原分支。

开发过程中 Mary 可以尽情提交 (commit)。如果提交历史看起来很混乱，完全可以使用 interactive rebase 命令移除或合并提交。这对于大规模项目意义重大，清晰的历史记录意味着项目维护者能够轻易读懂开发者的意图。

Mary 推送功能至远程仓库

开发完成后，Mary 使用 git push 命令将本地分支推送到自己的远程仓库（而非官方仓库）。

这使得其他开发人员（如项目维护者）能够访问她的代码。

Mary 发起 Pull Request

推送后，Mary 可以通过 Bitbucket 账户发起 Pull Request：找到自己的远程仓库， 点击右上角的 Pull Request 按钮。当前仓库默认为原仓库，Mary 需要指定原分支，目标仓库和目标分支。

要想将修改成为基础代码的一部分，Mary 需要将原分支指定为自己的功能分支，指定目标仓库是 john 的官方仓库， 目标分支指定为 master 分支。此外还需要填写标题和描述。除了 John 以外，Mary 还可以邀请将除了 John 以外的任何人审阅代码，只需将他们添加为 Reviewer 即可。

Pull Request 一旦创建，通知会以邮件和 Bitbucket 消息的形式发送给 John。

John 审阅 Pull Request

John 通过点击自己 Bitbucket 仓库的页面上的 Pull request 标签，查看所有的 Pull Request。再次点击 Mary 的 Pull Request，具体描述，提交记录及所有改动一览无余。

如果一切顺利，John 只需点击 Merge 按钮，将修改合并至 master 分支即可。

但假设 John 发现了一个小 bug，希望 Mary 在合并之前修复。那么他可以针对整个 Pull Request 或特定提交记录做出评论。

Mary 追加提交

对于任何疑问，Mary 都可以在 Pull Request 中反馈，将其作为讨论组使用。

Mary 在功能分支修改并提交，修正错误，推送到自己的远程仓库，正如她之前做的一样。这次提交被自动追加到之前的 Pull Request 中，显示在先前 John 的评论之后，便于再次审阅。

John 接受 Pull Request

终于，John 接受修改，将代码合并至 master 分支，关闭 Pull Request。至此新功能被正式添加到项目中，其他开发者可以通过 git pull 命令获取更新。

下一步

你已经掌握了使用 Pull Request 所需的一切知识，是时候将它融入自己的工作流程当中了。记住，Pull Request 不是 Git-based collaboration workflows 的代替品。而是在此之上，让团队协作更容易的催化剂。

Merge

  • 如果两个 commit 处于同一条历史上，没有分支，则二者将会以 fast-forward 的形式 merge，不产生新的 commit
  • 两个 commit 位于不同分支，则这是一个三方merge，tip1 + tip2 + common ancestor，产生新 commit，有两个parent
  • 发生冲突时，冲突的部分将会添加 marker，以便我们手动解决。解决后，应该通过 git add 添加到 staging area，表示冲突已解决（冲突其实就是 git 对文件做了修改，要求我们再次修改为正确的，文件的状态从 unmodified > modified > staged）
  • 将 base branch rebase 至 target branch
  • 不产生新的 commit
  • 结果就是所有 commit 都是线性的，没有分叉
  • rebase 后，target branch 和 base branch merge时，只需 fast-forward 即可
  • merge过后，base branch 可以删除
  • 绝对注意，只 rebase 私有分支，千万不要对公共分支操作

Rebase 的步骤

容器视图控制器

发表于 2019-06-26 | 分类于 开发

缘起

作为一名 iOS 开发者， “UIViewController 到底是干嘛的？” 这个问题一直让我感到困惑。说来惭愧，早已不是新手的我不但没能搞懂这个问题，更经常拿“会用就行，细节不重要”之类的话安慰自己，不求甚解，得过且过。直到最近，又一次因为项目需要，必须自行实现容器控制器。经过一番搜索，查阅之后，终有所感悟，特此记录。

组合

同类对象组合，形成特定结构，是 UIKit 中常见的设计模式。例如，UIView 组成 view hierarchy，UIResponder 组成 responder chain。类似的，视图控制器之间也可以互相组合，有两种方式：

  1. modal presentation - 基于当前视图，展示另一个控制器的视图，阻断下层视图的交互
  2. 容器控制器 - 一个控制器同时持有一个或多个子控制器，将它们的视图据为己用

说到这里，很多人会有一个疑问：为什么（要组合控制器）？有什么好处？如果单纯只是为了借用其他控制器的视图，直接加入当前视图结构不就行了？

然而，除开管理视图，控制器还有许多重要职责。其中最具代表性的，当属转发各种视图事件：

  • viewWillAppear
  • viewDidAppear
  • viewWillDisappear
  • viewDidDisappear

这些方法对于开发的重要性不言而喻。仅将视图加入当前视图结构，是不会触发这些方法的。

今天我们将探讨第二种组合方式：容器控制器。思考 UIKit 是如何实现这类控制器的；自己实现时如何组织思路，要注意哪些细节；最后，回答开篇的问题： “UIViewController 到底是干嘛的？” 。

UINavigationController

UINavigationController 是 UIKit 中最具代表性的容器控制器（当然还有 UITabBarController）。或许我们早已习惯它的存在，本着够用就好的原则，很少考虑其实现原理。换句话说，如果让你自行实现一个导航控制器，要解决哪些问题呢？例如：

  • 如何处理导航栏与子控制器关系的？
  • 如何触发子控制器的视图事件？
  • 两个子控制器之间如何过渡？
  • 如何实现手势返回？

当然，本文的目的不是回答上述问题。而是想强调没有什么黑魔法，只要我们愿意，一样可以实现功能极为复杂的容器控制器。

套路

首先必须明确，容器控制器的目标在于建立两套联系：

  1. 父控制器与子控制器之间的联系
  2. 视图结构

从功能出发，容器控制器必须能够添加、删除子控制器，这里给出如下套路，照搬即可：

添加

  1. 建立父子关系。注意，这个方法首先自动调用子控制器的willMoveToParentViewController:，通知其即将加入父控制器；
  2. 将子控制器视图加入自己的视图结构；
  3. 通知子控制器，它已加入父控制器；

删除

  1. 通知子控制器，其即将被移出父控制器；
  2. 将子控制器视图移出视图结构；
  3. 解除父子关系。注意，这个方法最后调用didMoveToParentViewController，通知子控制器父子关系已经解除。

转发视图事件

UIKit 不推荐直接调用子控制器的视图事件方法，而是使用 beginAppearanceTransition:animated: 和 endAppearanceTransition。我们需要基于自定义容器控制器的显示特点，在合适的地方调用这两个方法。显然，这将大大增加工作量。所幸，UIKit 提供 shouldAutomaticallyForwardAppearanceMethods 自动转发视图事件。 但其工作原理并不透明。手动转发 vs. 自动转发，孰优孰劣暂时无法给出结论， 有机会的我们将在另一片文章中比较。

视图控制器的责任

现在，我想回答最初的问题：”UIViewController 到底是干嘛的？”

搞清这个问题其实很简单：打开苹果文档，找到 UIViewController Reference，查看它的 topics，每个 topic 都是它的一项职责，包括但不限于：

  • segue - 转场
  • view events - 视图事件
  • modal presentation - 阻断式显示
  • safe area - 安全区域
  • rotation - 屏幕旋转
  • 3D touch
  • status bar
  • containment - 容器控制器

不难看出，上述职责或多或少都与显示有关，但又不是纯显示，所以下放给 UIView 是不合适的。能够履行这些职责，并根据要求操作视图的对象，才是合理人选。视图控制器这个名字，完美阐释了它的定位，苹果在设计模式方面的功底可见一斑。

URL 概念与比较

发表于 2019-04-23 | 分类于 代码

URL与URI

说到URL，必须从URI讲起。

出于 hpertext 的需要，人们需要一种准确的方法表示资源。URL和URN分别从各自的维度定义资源：获取的方式 / 名称。二者属于 URI 的不同子概念。

关于URL与URN的区别，简单来说就是：URN可以是一个人的身份证号；而URL是这个人的住址。

个人理解，相较于上述经典定义，目前对URL的使用已经升华为表示一个动作，如获取资源，执行任务（如查询，这是我们每天都会使用的）。

注意

不要纠结于URL与URI到底有什么区别，完全没有必要。根据W3C的最新定义，URL和URI广义上的意义相同，即代表资源的字符串。所以用哪个都行，但在编程语言中，前者使用较多。

下面讨论的各项URI的性质，都适用于URL。

语法

URI的完整语法形式如下：

  • scheme，规定URI剩余部分的语法。常见的scheme有如http / https / ftp / mailto / file / data / irc。注意，虽然大部分scheme都有同名protocol与之对应（这是其设计之初的用途），但例外是存在的，如file。
  • 双反斜线（//）。如果authority不存在，则path不应以包含//。
  • authority，可以由下述部分组成：
    • authentication，包含用户名和密码，用于验明身份；
    • host，可以是域名，ip地址或任意表示作用域的字符串；
    • 端口号，如果host是域名或ip地址时。
  • 路径，类似于文件路径（但不等同），以层级的形式定义，层级之间用/分隔，以/开头。（注意，根据这里的定义，加上之前的//，会变成///。这是正常的，但显示时，还是会变成//）
  • query，表示查询的内容。以?开头，单个查询的形式为key=value，多个查询用&分隔。
  • fragement，表示当前URI所指内容的一部分。以#开头。例如，如果当前URI指向一篇文章，则#Summary指这个文件的Summary部分。

URI中的字符可以是ASCII字符中的英文，数字，-，.，_，~。其他字符必须百分号转义。

注意

~在URI中是不需要转义，这意味着我们可以使用它定义path，代表文件路径。这种用法实际上来自Unix：~代表当前用户的home directory，是一种惯例。使用URL表示文件路径时，要格外注意。

绝对与相对

所谓绝对URI指具有完整形式的URI；相对URI指URI reference。

URI reference的形式包括但不限于：

  • 完整URI：http://example.org/wiki/Main_Page
  • 不包括scheme：example.org/wiki/Main_Page
  • 只有部分path：wiki/Main_Page
  • 空：nothing here -_-

使用时，根据特定算法（RFC 3986），将URI reference与base URI合并，从而得出一个完整URI。很明显，如果URI reference本身就是一个完整URI，则base URI不起作用。
